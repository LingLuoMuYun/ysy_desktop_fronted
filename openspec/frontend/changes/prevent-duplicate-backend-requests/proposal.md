# prevent-duplicate-backend-requests

## 背景

在审查前端与后端对接代码时，发现部分请求路径存在重复发送或请求风暴风险，可能导致智能体后端或环境后端压力异常，严重时表现为后端卡死或长时间无响应。

## 问题

- `chatApi.editLatestMessage()` 会先打开一次 `/api/chat/stream`，随后再调用 `sendMessage()` 发第二次 `/api/chat/stream`，第一条流没有消费也没有取消。
- 首页和右侧 AI 助手仅依赖 React state 的 `isStreaming` 阻止重复发送，双击或回车连发时可能在 state 更新前进入两次发送逻辑。
- `environmentsApi` 在环境列表 500 错误时会尝试最多 200 次 `pageSize=1` 分页恢复请求，后端异常时反而放大压力。

## 目标

- 阻止同一聊天入口在一轮请求未完成前重复发送。
- 移除未使用且会制造双请求的聊天编辑占位逻辑。
- 移除环境列表 500 后的高频恢复分页请求。
- 保持现有接口契约不变。

## 非目标

- 不新增后端接口。
- 不实现聊天编辑、重新生成、候选切换。
- 不修改运行环境接口字段契约。

## 风险等级

低风险。修改集中在前端请求防抖、防重入和异常降级，不新增高风险本机动作。

## 验收标准

- 快速双击首页发送按钮或回车连发时，只产生一轮聊天请求。
- 快速双击右侧助手发送按钮或回车连发时，只产生一轮聊天请求。
- 环境列表加载失败时只失败并提示，不再发起最多 200 次恢复分页请求。
- `pnpm run typecheck` 通过。
