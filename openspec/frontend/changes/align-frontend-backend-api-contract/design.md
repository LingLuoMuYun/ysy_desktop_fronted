# Design

## 接入原则

1. 后端接口文档优先于前端旧接口替换文档和 mock 字段。
2. 前端服务层统一解析 `success/data/error/requestId` 响应包裹。
3. 页面组件不直接拼接请求细节，统一通过 service 或 feature hook 调用。
4. 所有改变本机状态或产品记录的动作必须走确认流程。
5. 可能重复提交的接口必须传 `Idempotency-Key` 或后端认可的 `idempotencyKey`。
6. 未在后端文档列出的接口标记为待确认，不新增临时接口。

## 分阶段方案

### Phase 1: 契约与客户端基础

- 建立统一 API client：base URL、请求包裹、错误映射、取消、超时。
- 建立通用类型：分页、统一响应、统一错误、检查结果、风险确认、幂等键。
- 定义状态 code 到 UI 文案和 tone 的映射，不再让页面直接依赖中文状态。

### Phase 2: 低风险读接口

- 首页工作台、本机资源、最近操作。
- 项目、任务、数据、模型、环境列表和详情。
- 规格接口：项目规格、任务规格、部署测试规格、安装初始化状态等。

### Phase 3: 中风险记录类写接口

- 创建项目、创建任务、导入数据集、导入模型、绑定项目、保存个人信息和偏好。
- 提交前展示摘要确认，后端仍校验 `confirmed`。

### Phase 4: 高风险动作接口

- 清理本地文件、启动/停止/删除任务、删除项目/数据/模型/环境、修复环境、依赖安装、模型下载和取消。
- 统一确认弹窗必须展示影响对象、是否启动/停止进程、是否修改文件或环境、是否可撤销。

### Phase 5: 页面替换与验收

- 按模块替换 mock：设置/环境已接入部分优先核对，再做首页、项目、任务、数据、模型。
- 每个模块补齐 loading、empty、error、confirm、retry、cancel、offline/local 状态。
- 只在真实接口或可控 mock adapter 下验证，不把 mock 当作真实后端验收。

## 需要明确的契约差异

- 模型模块后端新增 5.8-5.13 模型下载接口，前端旧索引未完整覆盖。
- 安装初始化 6.1-6.12 是新版后端新增完整模块，前端需要确认入口和页面归属。
- 项目编辑接口未在新版后端项目章节中列出，不能按旧 `PATCH /api/projects/{projectId}` 实现。
- AI 模型配置接口是否继续由智能体后端维护，需要与设置页 AI 配置边界保持一致。
