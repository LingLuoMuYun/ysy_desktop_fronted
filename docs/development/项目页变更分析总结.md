## 项目总体大纲

桌面智算是面向本地深度学习实验和轻量大模型工作流的桌面应用。项目整体围绕“本机 AI 工作台”展开，核心目标不是做云端训练平台，而是在个人电脑、本地工作站或小型实验环境中，把项目、数据、模型、运行环境、训练任务、推理服务和 AI 辅助诊断组织成一条可追踪的本地工作流。

当前 V1.0 的主线可以概括为：

```text
首页对话 / 快捷入口
  -> 创建或选择项目
  -> 绑定数据、模型、运行环境
  -> 创建训练任务或部署任务
  -> 运行、查看日志、诊断失败
  -> 沉淀模型资产和报告
  -> 通过 AI 助手解释、辅助填写和生成建议
```

六个一级模块的定位如下：

| 模块 | 核心职责 | 与项目页关系 |
|------|----------|--------------|
| 首页 | 对话式工作台、本机资源概览、快速入口 | 可发起创建项目、分析当前项目和查看最近操作 |
| 项目 | 长期实验容器，保存项目上下文和资源关系 | 项目页是任务、数据、模型、环境之间的业务枢纽 |
| 任务 | 训练任务和部署任务的执行、状态、日志和结果 | 任务必须归属项目，项目详情应能查看任务记录 |
| 数据 | 数据集导入、检查、绑定和质量分析 | 项目通过数据集绑定获得训练前置条件 |
| 模型 | 模型资产导入、检查、登记和推理服务建议 | 项目详情应能查看项目下模型资产 |
| 设置 | 运行环境、AI 助手模型和偏好配置 | 项目创建和任务创建需要引用运行环境 |

项目页在这套结构中承担“上下文入口”的角色。用户不应该在项目页直接配置训练脚本、启动命令或输出目录，这些执行快照属于任务模块；项目页需要提供的是项目长期信息、资源关系和进入任务/模型/数据流的入口。

## 前端总体架构

前端采用 React + TypeScript + Vite + Electron。整体结构按 Page、Feature、Service、Layout、Component 分层：

```text
frontend/src/
  app/                 # 应用入口和路由装配
  layouts/             # AppShell、Sidebar、WindowTitleBar、AssistantPanel
  pages/               # 一级页面装配层
  features/            # 按业务域拆分的局部流程和组件
  services/            # API / IPC 请求封装
  components/          # 可复用展示组件
  types/               # 前端领域类型
  styles/              # 全局样式和设计 Token
  mocks/               # 原型和 fallback 数据
```

当前项目页遵循以下前端调用方向：

```text
ProjectsPage
  -> features/projects/*
      -> services/projectsApi
          -> Vite proxy / 后端业务 API
  -> components/*
  -> layouts 提供的全局工作台环境
```

关键边界：

- Page 负责页面级装配，不直接承载完整业务流程。
- Feature 负责单个业务域内的表单、弹窗、Hook 和局部状态。
- Service 负责后端 DTO 适配、请求发送、错误信息映射和回退地址。
- Component 只做展示和局部交互，不直接请求后端。
- Layout 负责三栏工作台、顶部栏、右侧助手等全局结构。

这种拆分的直接收益是项目详情、创建项目、删除项目和列表加载可以独立迭代。后续接入任务记录、模型资产和操作记录时，可以继续在 `features/projects/` 内扩展 Hook 或子组件，而不把 `ProjectsPage.tsx` 重新堆成大组件。

## 项目功能体系

### 1. 项目列表

项目列表是用户进入项目上下文的主入口。当前具备：

- 项目类型筛选。
- 项目状态筛选。
- 项目名称 / 路径搜索。
- 分页。
- 项目卡片展示。
- 创建项目入口。
- 删除项目入口。
- 项目详情入口。

项目卡片展示字段围绕“能否继续工作”组织：

- 名称和状态用于快速判断项目当前阶段。
- 描述用于说明项目目标。
- 路径用于定位本机工作空间。
- 训练任务、部署任务、数据集、模型数量用于展示资源沉淀。
- 更新时间用于判断最近活跃程度。

### 2. 项目创建

项目创建只保存长期上下文：

- 项目名称。
- 项目类型。
- 项目描述。
- 项目保存位置。
- 默认运行环境。

当前不在项目创建中要求：

- 训练脚本。
- 启动命令。
- 工作目录。
- 输出目录。
- 依赖文件。
- 初始数据集绑定。

原因是项目不是执行配置本身。训练脚本、命令、输出目录等属于一次具体任务的执行快照，如果放在项目层，会导致项目和任务边界混乱，也会让后续复制任务、复现实验和诊断失败变得不清晰。

### 3. 项目删除

删除项目按高风险操作处理。当前前端请求明确传入：

```json
{
  "confirmed": true,
  "deleteLocalFiles": false
}
```

该设计保证删除只影响产品内登记记录，不删除本机文件、数据集、模型权重或训练产物。未来如果支持删除本地文件，必须增加清理预览、路径列表和二次确认。

### 4. 项目详情

项目详情弹窗是项目上下文的聚合入口，当前包含：

- 项目概览。
- 操作记录。
- 任务记录。
- 模型资产。

当前只接入项目基础详情，后三个 Tab 为明确空态。该处理避免在没有接口联调时伪造数据，也让后续接入边界清楚：

- 操作记录复用活动记录接口或后续项目操作记录接口。
- 任务记录复用任务列表接口。
- 模型资产复用模型列表接口。

### 5. 项目与 AI 助手

项目页右侧仍由全局 `AppShell` 渲染 AI 助手面板，页面内部不重复嵌套助手。项目相关对象后续可作为 AI 上下文传入，包括：

- 当前项目概览。
- 某个任务记录。
- 某个模型资产。
- 某次操作记录。
- 创建任务或部署任务表单字段。

AI 在项目页的边界应保持三类行为：

- `explain`：解释项目字段、状态、风险和下一步。
- `field_suggestion`：给出表单建议，由用户确认后写入。
- `action_request`：请求创建、启动、删除等真实动作，必须走页面入口和后端确认机制。

## 重要技术模块

### 1. `features/projects/useProjectList.ts`

该 Hook 是项目列表的状态中心，负责：

- 从 URL 恢复筛选条件。
- 管理 `keyword`、`selectedType`、`selectedStatus`、`currentPage`。
- 请求项目规格和项目列表。
- 筛选条件变化时重置分页。
- 将筛选条件写回 URL。
- 暴露 `reload` 给页面重试或创建成功后刷新。

它避免了页面组件同时维护筛选、分页、请求和 URL 同步。

### 2. `services/projectsApi.ts`

项目服务层负责前后端契约适配：

- `list`：项目分页列表。
- `specs`：项目类型、状态选项、工作区建议和可用环境。
- `detail`：项目基础详情。
- `create`：创建项目并传入 `confirmed=true`。
- `delete`：删除项目登记并固定 `deleteLocalFiles=false`。
- `validateWorkspace`：校验项目保存位置。

重要处理：

- 支持 `VITE_PROJECTS_API_BASE_URL` 覆盖项目后端。
- 开发环境下可回退到同源 `/api/projects` 代理。
- 将后端 `status/statusText` 映射为前端展示状态和 `StatusTone`。
- 将 `datasetCount` 统一为数字，展示单位留给 UI 层。

### 3. `ProjectCreateDialog`

项目创建采用“两步确认”：

1. 表单填写和路径校验。
2. 确认摘要和正式提交。

这符合中风险操作的要求：新增产品内项目记录前，用户需要看到项目名称、类型、保存位置和默认环境摘要。

### 4. `ProjectDeleteDialog`

删除项目采用高风险确认弹窗，强调：

- 影响对象。
- 本地文件不删除。
- 是否可撤销取决于后端恢复能力。
- 请求将传入 `confirmed=true`。

当前弹窗没有提供删除本地文件选项，避免误删用户数据。

### 5. `ProjectDetailDialog`

项目详情弹窗负责项目上下文展示：

- 使用项目列表项作为 fallback，详情接口失败前仍能展示基础入口状态。
- Tab 切换为本地 UI 状态，不触发未实现接口。
- 项目概览展示来自详情接口或 fallback 的字段。
- 操作记录、任务记录、模型资产保留空态，明确后续复用对应模块接口。
- 弹窗宽度、字号和按钮尺寸按桌面工作台密度收敛，并补充窄宽度自适应。

### 6. `ProjectsPage.css`

项目页样式处理了两个关键问题：

- 页面级样式使用 `.workbench-page.project-page` 覆盖全局 `.workbench-page`，避免 CSS 特异度和加载顺序导致布局失效。
- 列表容器使用 Flex + `min-height: 0`，保证滚动区域填满剩余空间，并让项目卡片从顶部排列。

弹窗样式重点处理：

- 自适应宽度。
- 内容区最大高度。
- Tab 横向滚动。
- 底部按钮在窄宽度下换行。

### 7. `WindowTitleBar` 刷新反馈

顶部栏刷新按钮增加短时旋转动画和 600ms 禁用状态。该改动不改变业务刷新逻辑，只补充点击反馈，降低连续点击造成的重复触发概率。

### 8. Vite 代理

`vite.config.ts` 补充 `/api/projects` 代理，让开发环境项目接口可以走同源转发。该配置与 `projectsApi` 的回退策略配合，降低浏览器直接访问业务后端时的 CORS 和网络环境差异影响。

## 变更概览

### 1. 项目页结构拆分

`ProjectsPage.tsx` 从大组件拆为页面装配层，项目创建、删除、详情和列表状态逻辑下沉到 `frontend/src/features/projects/`。

当前拆分结果：

- `useProjectList.ts`：负责项目列表加载、筛选、分页和 URL 状态同步。
- `ProjectCreateDialog.tsx`：负责创建项目两步流程，包含路径校验和确认提交。
- `ProjectDeleteDialog.tsx`：负责删除项目高风险确认。
- `ProjectDetailDialog.tsx`：负责项目详情弹窗。
- `projectFormHelpers.tsx`：复用项目表单字段、确认信息行和状态占位。

收益：

- 页面组件只负责组合和桥接 API / Toast / Dialog 状态。
- 创建、删除、详情弹窗可以独立维护。
- 后续接入项目详情内的任务记录、模型资产和操作记录时，改动边界更清晰。

### 2. 项目列表布局修复

修复了项目列表在单卡片或少量卡片时高度无法填满页面剩余空间的问题。

关键处理：

- 将页面级选择器从 `.project-page` 改为 `.workbench-page.project-page`，提高特异度，避免被全局 `.workbench-page` 的 Grid 布局覆盖。
- 将项目页改为纵向 Flex 布局。
- `.project-list-shell` 和 `.project-grid` 使用 `flex: 1`、`min-height: 0` 和顶部对齐。
- 移除会导致列表壳体按内容高度收缩的额外上边距与边框干扰。

结果：

- 项目列表壳体从搜索栏下方延展到页面底部。
- 单项目卡片不再贴近分页栏底部。
- 卡片从滚动区域顶部顺序排列。

### 3. 项目详情弹窗

新增项目详情弹窗交互。

交互入口：

- 点击项目名称。
- 点击项目卡片空白区域。

弹窗内容：

- 头部：项目图标、项目名称、状态标签。
- Tab：项目概览、操作记录、任务记录、模型资产。
- 项目概览：项目名称、项目类型、项目状态、项目保存位置、默认运行环境、更新时间。
- 底部：创建部署任务、创建训练任务入口占位。

自适应优化：

- 弹窗最大宽度收敛为 680px。
- 使用 `max-height` 随视口收缩，避免固定大高度。
- 标题、字段、Tab 和按钮字号下调。
- 小宽度下标题可换行，Tab 可横向滚动，底部按钮可换行。

当前未接入：

- 操作记录列表。
- 任务记录列表。
- 模型资产列表。
- 底部创建任务入口真实跳转或表单打开。

### 4. 项目 API 与类型适配

项目接口层调整：

- 项目后端默认地址更新为 `http://10.0.221.143:8000`。
- Vite 代理补充 `/api/projects` 到业务后端。
- `ProjectSummary.datasetCount` 从字符串改为数字。
- Mock 项目数据同步改为数字类型。

收益：

- 类型更贴近后端接口契约。
- 列表中数据集数量和后续统计计算更容易复用。
- 本地开发时项目接口可通过 Vite proxy 转发到业务后端。

### 5. 顶部刷新按钮反馈

`WindowTitleBar` 的刷新按钮增加短时旋转反馈。

实现方式：

- 增加 `isRefreshing` 状态。
- 点击后触发 `onRefreshPage()`。
- 按钮禁用 600ms，避免连续点击。
- `globals.css` 增加 `icon-spin` 动画。

收益：

- 用户点击刷新后有可见反馈。
- 降低短时间重复点击概率。
